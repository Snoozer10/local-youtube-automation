# Task 10 Report: End-to-End Validation (116-clip run)

## Summary
Ran `compile_video.py` (single-pass default, auto QSV) end-to-end on the real 116-clip
dataset `youtube_runs/Everyday Habits That Boost Brain Power/`. **The first attempts failed
in four distinct ways that real verification surfaced** (none caught by unit tests alone).
All four were fixed; the final render succeeded and the output meets the acceptance criteria.

## Bugs Found & Fixed During E2E (compile_video.py)
1. **Double UTF-8 BOM in subtitles** — `fix_arabic_srt` read input as `utf-8` (input already
   had a BOM) and wrote `utf-8-sig` (added a second BOM). ffmpeg's SRT demuxer couldn't parse
   the double BOM → `Unable to open timestamped_transcript_fixed.srt`. Fixed: read `utf-8-sig`
   (strips any BOM), write plain `utf-8` (no BOM).
2. **Relative image/audio paths vs `cwd=run_folder`** — `run_single_pass` invokes ffmpeg with
   `cwd=run_folder`, but `images_dir`/`audio_path` are relative to the project root, so every
   `-i youtube_runs/.../00_00.png` failed. Fixed: make `images_dir` and `audio_path` absolute
   inside `run_single_pass` (subtitle keeps basename + cwd, which works).
3. **Windows command-line length limit** — the 116-clip Ken Burns filtergraph exceeded the
   ~32K Windows CreateProcess limit → `WinError 206`. Fixed: write the graph to
   `filter_complex.txt` and use `-filter_complex_script` (path made absolute).
4. **Output path relative to `cwd`** — `output_path` was relative, so ffmpeg wrote to
   `run_folder/youtube_runs/.../youtube_ready_video.mp4`. Fixed: `output_path` made absolute.
5. **concat SAR mismatch** — clips from source images with differing aspect ratios produced
   different SARs, so `concat` aborted (`SAR ... do not match`). Fixed: every Ken Burns branch
   now ends with `,setsar=1,format=yuv420p` so all concat inputs are uniform.
6. **Render timeout too low** — `FFMPEG_FINAL_TIMEOUT=600` killed the QSV render (took ~27 min
   / 1616s). Raised to `5400` (config + DEFAULTS) to fit the 116-clip software-filter render.

## Minor Deviations (non-blocking)
- **Output pix_fmt is `yuvj420p`**, not `yuv420p`. h264_qsv rejects `yuv420p` ("auto-selecting
  nv12") and the resulting full-range 4:2:0 is reported as `yuvj420p`. Playback-compatible; noted
  for a future explicit `-color_range`/format pass if strict `yuv420p` is required.
- **Legacy checkpoint-resume path (`ENABLE_SINGLE_PASS=false`)** not executed (a full 116-clip
  CPU/legacy render would take far longer than available). Code review confirms it is unaffected
  by bugs 2–4: legacy clip rendering uses the default cwd (project root) so relative image paths
  resolve, and it already benefits from the `fix_arabic_srt` BOM fix. The single-pass path is the
  configured default and is what was validated.
